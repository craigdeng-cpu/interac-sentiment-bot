# Interac Intelligence Bot - Full System Overview

## 1) What This Project Is

This is a Python Telegram bot that continuously scans public web chatter related to Interac and adjacent payment competitors, then uses Moonshot Kimi to transform raw mentions into structured intelligence reports.

Primary job:
- collect public mentions (news/forums/social-indexed pages),
- enrich and cluster those mentions,
- run LLM analysis in different report modes,
- deliver results to Telegram and optional email dashboards.

The latest direction emphasizes a **fact-only, evidence-backed brandscan**:
- explicit dates,
- corroboration labels (`strong`, `moderate`, `early`),
- exact URLs or explicit `insufficient evidence`,
- separation of signal content vs signal quality,
- scan-to-scan movement summary.

- **Language/runtime**: Python 3.12
- **Bot framework**: `python-telegram-bot` (with job queue)

  - Resend API (`httpx`)

  - `Dockerfile` for container builds

## 4) End-to-End Runtime Flow

## 4.1 Boot
1. `app.py` reads required env vars (`TELEGRAM_TOKEN`, `KIMI_API_KEY`, etc.).
2. `main()` initializes Telegram `Application`.
3. Command/message handlers are registered.
4. Job queue schedules:
   - sentiment scans at 6am/10am/2pm/6pm EST
   - weekly digest at configured EST day/hour (implemented as daily job + in-callback guard).
5. Bot runs in polling mode unless `WEBHOOK_URL` is set (then webhook mode).

## 4.2 Data collection
Search is done through DuckDuckGo (`ddgs`) via `web_search()` and `search_twitter()` wrappers.

Main fetch paths:
- `fetch_all_mentions()` -> daily/persona scan corpus
- `fetch_historical_mentions()` -> `/deepscan` corpus
- `fetch_brand_archetype_mentions()` -> `/brandscan` corpus

During fetch:
- query groups from `prompts.json` are executed,
- results are de-duplicated,
- source metadata is extracted (channel/source quality),
- payload text is assembled for the LLM.

For brandscan specifically:
- mentions are enriched with brand/use-case/date metadata,
- rule-based story clustering groups similar mentions,
- corroboration and signal metrics are computed,
- `=== STORY CLUSTERS ===` and `=== SIGNAL QUALITY METRICS ===` are injected into LLM input.

## 4.3 Analysis
`call_kimi()` sends:
- system prompt from the selected prompt file,
- user content (mentions payload),
- configured model + generation params.

Analysis functions:
- `analyze_sentiment()`
- `analyze_personas()`
- `analyze_historical()`
- `analyze_brand_archetypes()`

`analyze_scan()` dispatches signal vs persona mode.

## 4.4 Brandscan post-processing (critical)
After Kimi returns brandscan output, `_postprocess_brandscan_report()`:
- normalizes/sanitizes claim lines,
- enforces date/corroboration/source fields where missing,
- rejects placeholder source tokens (maps to `Sources: [insufficient evidence]`),
- sorts claim lines by corroboration strength,
- compares current claims with prior snapshot (`state/brandscan_snapshot.json`),
- inserts `WHAT CHANGED SINCE LAST SCAN` section (`New`, `Strengthened`, `Weakened`, `Dropped`),
- persists new snapshot.

## 4.5 Delivery
- Telegram:
  - `send_chunked_message()` handles message length constraints and markdown fallback.
- Email:
  - `send_email()` routes to SMTP or Resend.
  - `build_email_bodies()` auto-detects report mode and chooses HTML renderer:
    - daily template
    - historical template
    - persona template
    - brand archetype template
    - raw fallback if parse markers are missing.

## 5) Bot Commands and What They Do

## Public commands
- `/start` or `/help`: intro + command list; auto-subscribes current chat.
- `/subscribe`: subscribe chat to scheduled pushes.
- `/unsubscribe`: stop scheduled pushes.
- `/status`: bot health/config summary.
- `/scan`: run immediate signal scan.
- `/scan persona`: run immediate persona scan.
- `/personas`: alias for `/scan persona`.
- `/raw`: show latest raw mentions.
- `/prompt`: show query/source/threshold summary.
- plain text message: follow-up Q&A against latest report context (rate-limited).

## Admin-only commands
- `/email`: run fresh daily signal scan and email it.
- `/deepscan`: run historical multi-window scan + email.
- `/brandscan`: run brand archetype scan + email.
- `/stop`: cancel active running jobs/tasks.
- `/smtpcheck`: verify email provider connectivity/config.

## 6) Report Modes

## Daily signal scan
- Primary output: sentiment score + short operational intelligence.
- Used by scheduled broadcasts and `/scan` (default mode).

## Persona scan
- Same raw mention ingestion path as daily scan.
- Different prompt contract focused on archetypes of users and pain clusters.

## Historical deep scan
- Multi-time-window long-horizon summary.
- Optimized for trend recurrence and thematic persistence.

## Brand archetype scan (most advanced pipeline)
- Uses broader competitor-leaning historical query set.
- Enrichment + clustering + corroboration pre-processing before LLM.
- Fact-only contract with evidence-first formatting and explicit signal quality.
- Includes scan-to-scan movement summary using snapshot state.

## 9) Scheduling and Timeouts

Scheduled jobs:
- sentiment broadcasts: 4x/day EST
- weekly digest: configured EST day/hour

Notable timeout controls:
- `/scan` fetch + analyze each guarded by ~120s
- `/deepscan` fetch ~180s, analyze ~120s
- `/brandscan` fetch ~210s (with budgeted partial-data fallback), analyze ~120s

Brandscan timeout resilience:
- fetch budget and partial data behavior reduce hard failures under heavy search load.

## 11) Current Strengths

- Single-file operational simplicity (easy to deploy quickly)
- Multiple intelligence modes from one ingestion layer
- Clear command surface for operators
- Fact-only direction for brandscan with corroboration semantics
- Email + Telegram dual-channel output
- Built-in task cancellation and diagnostics

---

## 12) Current Limitations / Risks

- Core state mostly in-memory (durability issues across restart/redeploy)
- Single large `app.py` file (high coupling, harder onboarding/testing)
- Limited automated test coverage (manual validation heavy)
- Search quality and URL/date completeness depend on upstream indexed data
- Linter import warnings in local IDE if dependencies/env not fully installed
- Snapshot comparison currently local-file based (not multi-instance safe)

   - optionally drop all claims lacking real URLs instead of retaining with `insufficient evidence`.

## 14) Quick "How It Works" Narrative (for new owner)

When a scan runs, the bot executes configured search queries, collects public mention snippets/URLs/dates, enriches them with metadata (brand/use-case/source quality), and sends that corpus to Kimi with a strict prompt template. Kimi returns a structured report. For brandscan, the app then post-processes the report to enforce claim hygiene and compare against the previous scan snapshot. Final output is delivered to Telegram and optionally rendered into email cards for stakeholders.
